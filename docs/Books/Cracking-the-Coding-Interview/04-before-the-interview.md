# Chapter IV — Before the Interview

> Acing an interview starts years before the interview itself. The right experience leads to the right resume, which leads to the right interviews. If you're starting late, do as much catching up as you can — then focus on preparation.

---

## Getting the Right Experience

Without a great resume, there's no interview. Without great experience, there's no great resume. Start building your profile as early as possible.

### For Current Students

| Strategy | Why It Works | How to Execute |
|----------|-------------|----------------|
| **Take big project classes** | Practical experience before formal work experience | Seek courses with real-world projects — the more relevant, the better |
| **Get internships early** | Each internship paves the way for a better one | Apply to freshman/sophomore programs at top companies; consider startups for flexibility |
| **Start something** | Shows initiative and builds real skills | Side projects, hackathons, open source contributions — the topic matters less than the act of building |
| **Contribute to open source** | Experience with large codebases, collaboration, code review | Pick a project you use, fix a bug, submit a PR — start small |
| **Build a portfolio** | Tangible proof of your abilities | GitHub profile with clean READMEs, deployed projects, consistent commit history |

### For Professionals

| Strategy | Why It Works | How to Execute |
|----------|-------------|----------------|
| **Shift work toward coding** | Builds resume-worthy experience without revealing job search | Discuss eagerness for bigger coding challenges with your manager; volunteer for "meaty" projects |
| **Use nights and weekends** | Side projects demonstrate passion and current skills | Build a mobile app, web app, or desktop tool — great for learning new technologies too |
| **Contribute to open source** | Shows community engagement and code quality | Even small contributions count — documentation, tests, bug fixes |
| **Take on leadership** | Positions you for senior roles and management tracks | Lead a project, mentor a junior developer, run a tech talk series |
| **Learn new technologies** | Keeps you relevant and shows growth mindset | Pick technologies used by your target companies — build something real with them |

> **The two things that matter**: Companies want to see that you're smart and that you can code. Everything on your resume should reinforce these two signals.

### Building the Right Project Portfolio

```
                     PROJECT IMPACT MATRIX

                 Low Technical Depth    High Technical Depth
              ┌─────────────────────┬─────────────────────┐
  High        │  Tutorial projects  │  ★ SWEET SPOT ★     │
  Visibility  │  (less impressive)  │  Deployed apps,     │
              │                     │  open source with   │
              │                     │  real users          │
              ├─────────────────────┼─────────────────────┤
  Low         │  Skip these         │  Good foundation    │
  Visibility  │                     │  (algorithms,       │
              │                     │  data structures     │
              │                     │  from scratch)       │
              └─────────────────────┴─────────────────────┘
```

Aim for 2–4 strong projects that demonstrate:
- **Solving a real problem** (not just following a tutorial)
- **Technical depth** (non-trivial architecture, interesting algorithms)
- **Completion** (deployed, usable, documented)
- **Your role** (especially if it was a team project)

## Writing a Great Resume

Resume screeners look for the same things interviewers do: **smart + can code**. Your resume has about 10 seconds to make that case.

### The One-Page Rule

| Experience | Recommended Length |
|-----------|-------------------|
| < 10 years | **1 page** (strongly advised) |
| 10+ years | 1.5–2 pages maximum |

> Shorter resumes are often more impressive. Recruiters spend ~10 seconds scanning. If you limit content to your most impressive items, they're guaranteed to see them. Long resumes aren't a sign of experience — they're a sign of poor prioritization.

### Resume Structure

```
┌────────────────────────────────────────────────┐
│  YOUR NAME                                      │
│  email | phone | github | linkedin | portfolio  │
├────────────────────────────────────────────────┤
│  EDUCATION                                      │
│  University, Degree, Graduation Year, GPA (if   │
│  strong — 3.5+), relevant coursework            │
├────────────────────────────────────────────────┤
│  EXPERIENCE                                     │
│  Company — Role — Dates                         │
│  • Accomplished X by implementing Y, leading    │
│    to Z (measurable result)                     │
│  • Repeat for 2–4 strongest bullets             │
├────────────────────────────────────────────────┤
│  PROJECTS                                       │
│  Project Name — Technologies Used               │
│  • Brief description + your role + impact       │
│  • 2–4 most significant projects only           │
├────────────────────────────────────────────────┤
│  TECHNICAL SKILLS                               │
│  Languages: Java (expert), Python (proficient), │
│  C++ (prior experience)                         │
│  Frameworks: Spring, React, ...                 │
│  Tools: Docker, AWS, Git, ...                   │
└────────────────────────────────────────────────┘
```

### Writing Strong Bullets

Use the **"Accomplished X by implementing Y which led to Z"** formula:

| Weak Bullet | Strong Bullet |
|-------------|---------------|
| "Worked on the search team" | "Reduced search latency by 40% by implementing an inverted index with caching, serving 2M daily queries" |
| "Responsible for backend services" | "Designed and deployed 3 microservices handling 50K RPM, reducing order processing time from 5s to 800ms" |
| "Built a recommendation engine" | "Increased user engagement by 25% by building a collaborative filtering recommendation engine processing 10M user interactions daily" |
| "Fixed bugs in the Android app" | "Reduced crash rate from 2.1% to 0.3% by identifying and fixing 15 memory leak patterns in the Android app" |

### Key Principles

| Principle | Details |
|-----------|---------|
| **Show impact, not responsibilities** | "Responsible for" is weak. "Reduced/Increased/Designed/Built" is strong |
| **Quantify everything** | Percentages, user counts, latency numbers, team sizes — make results measurable |
| **Include only relevant positions** | Your resume is not a complete employment history |
| **Prioritize projects** | 2–4 significant projects. Independent projects beat course projects. Don't add all 13 |
| **Be honest about languages** | List proficiency levels: "Java (expert), C++ (proficient), JavaScript (prior experience)" — anything on your resume is fair game |
| **Avoid years of experience** | "10 years of Java" is ambiguous. Proficiency descriptors are clearer |

### Employment History

Your resume should **not** include a full history of every role. Include only positions that make you a stronger candidate. A 3-year stint doing non-technical work at a non-tech company typically doesn't help.

### Projects Section

| Do | Don't |
|----|-------|
| List 2–4 most significant projects | List all 13 projects you've ever done |
| State technologies used | Leave technologies vague |
| Note individual vs team project (if it helps) | Claim team work as individual if asked about it |
| Prefer independent projects over course projects | List homework assignments |
| Highlight projects with real users/impact | List half-finished prototypes |

## Language and Technology Pitfalls

### Potential Stigmas to Be Aware Of

| Stigma | Why It Exists | How to Mitigate |
|--------|--------------|-----------------|
| **Enterprise languages (VB, COBOL)** | Associated with less sophisticated applications | Showcase modern projects alongside; demonstrate breadth |
| **.NET ecosystem bias** | Silicon Valley bias toward open-source stacks | Highlight problem-solving ability over any specific platform |
| **"Language flaunting"** | Listing every Java version signals specialization over versatility | Focus on proficiency levels, not version numbers |
| **Only 1–2 languages** | Suggests limited exposure to different paradigms | Learn and build projects in diverse languages (e.g., Java + Python + Go) |
| **Certifications** | Some companies see them as a proxy for "enterprise" culture | Include them if the target company values them; omit for FAANG-style companies |

### Recommended Language Diversification

```
Instead of:                     Aim for:

Java, Java EE, Java SE,  ───→  Java (expert), Python (proficient),
Spring Boot, JUnit              Go (prior experience)

Pick languages from different paradigms:
  • OOP:        Java, C++, C#
  • Scripting:  Python, Ruby, JavaScript
  • Systems:    Go, Rust, C
  • Functional: Scala, Haskell, Clojure
```

### Resume Tips for Non-Native English Speakers

1. **Get a native English speaker to proofread** — some companies discard resumes with typos
2. For US positions, **do not include** age, marital status, nationality, or a photo
3. Use action verbs and the X-by-Y-leading-to-Z formula regardless of language background

## Study Plan by Time Available

| Time Available | Daily Commitment | Problems Goal | Focus Areas |
|---------------|-----------------|---------------|-------------|
| **< 2 weeks** | 4–6 hours | 30–50 | Big O, Arrays, Trees, most common patterns |
| **1 month** | 2–3 hours | 80–120 | Add linked lists, stacks/queues, recursion/DP, sorting |
| **3 months** | 1–2 hours | 150–200 | Full coverage including system design, bit manipulation, math |
| **6+ months** | 1 hour | 200+ | Deep practice, mock interviews, system design mastery |

### Problem Difficulty Distribution

```
Easy    ████████████ 30%     ← Build confidence, learn patterns
Medium  ████████████████████ 50%     ← Core interview difficulty
Hard    ████████████ 20%     ← Stretch goals, competitive edge
```

> Focus on **medium** problems. They represent the vast majority of actual interview questions. Easy problems build confidence. Hard problems give you an edge when you encounter them, but don't spend all your time here.

## Modern Preparation Strategies

### Online Platforms

| Platform | Best For |
|----------|---------|
| **LeetCode** | Largest problem set, company-tagged questions, discussion forums |
| **HackerRank** | Timed challenges, company-specific assessments (Amazon, Palantir) |
| **CodeSignal** | Company assessment platform, General Coding Score |
| **Pramp** | Free mock interviews with real people |
| **interviewing.io** | Anonymous mock interviews with engineers from top companies |
| **Neetcode** | Curated problem lists organized by pattern (Blind 75, Neetcode 150) |
| **AlgoExpert** | Video explanations + curated problem list |

### LinkedIn Optimization

1. **Headline**: Include target role + key technologies (e.g., "Software Engineer | Java, Distributed Systems, AWS")
2. **About section**: Brief narrative of your experience, interests, and what you're looking for
3. **Experience**: Mirror your resume's strong bullets — recruiters check both
4. **Skills & endorsements**: List your strongest technical skills; ask colleagues to endorse
5. **Open to Work**: Use the private setting visible only to recruiters
6. **Engage**: Comment on technical posts, share articles — visibility helps

### Portfolio Strategies

| Component | Purpose | Tips |
|-----------|---------|------|
| **GitHub profile** | Showcase code quality and activity | Clean READMEs, consistent commits, pinned repos |
| **Personal website** | Professional presence, blog, project showcases | Keep it simple and fast; content > design |
| **Blog / Technical writing** | Demonstrate communication and depth | Write about problems you've solved, systems you've built, concepts you've learned |
| **Open source contributions** | Show collaboration skills | Even small contributions (docs, tests, bug fixes) count |

### Latency Numbers Every Programmer Should Know

| Operation | Time |
|-----------|------|
| L1 cache reference | 0.5 ns |
| L2 cache reference | 7 ns |
| Main memory reference | 100 ns |
| SSD random read | 150 μs |
| HDD seek | 10 ms |

### Common Complexity Classes

| Complexity | Name | Example |
|-----------|------|---------|
| O(1) | Constant | Hash table lookup |

## Actionable Takeaways

1. **Start early** — building the right experience takes months or years, not weeks
2. **Your resume is a one-page highlight reel** — cut ruthlessly, keep only what's impressive
3. **Use the X-by-Y-leading-to-Z formula** for every bullet on your resume
4. **Practice on paper or a whiteboard** — IDE problem-solving is a different skill
5. **Track your mistakes** — a mistake list is one of the most effective study tools
6. **Mock interviews are non-negotiable** — solving problems alone is not the same as under pressure with an audience
7. **Don't cram the day before** — trust your preparation and get a good night's sleep
8. **Follow up politely** — silence is normal, not a signal
9. **Rejection is not failure** — it's information. Learn from it and try again
