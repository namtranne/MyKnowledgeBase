рҹ”Қ Indexing & Query Optimization

Indexes are the single most impactful tool for database performance. Understanding how they work internally вҖ” not just how to CREATE INDEX вҖ” is what separates senior engineers in interviews.

рҹҢі How Indexes WorkвҖӢ

An index is a separate data structure that maintains a sorted mapping from key values to row locations (page + offset), enabling the database to find rows without scanning the entire table.

Lookup Flow: SELECT * FROM users WHERE id = 45вҖӢ

Step 1:  Root node вҶ’ 45 > 30, 45 < 60 вҶ’ go to middle child
Step 2:  Internal [40|50] вҶ’ 45 > 40, 45 < 50 вҶ’ go to middle child
Step 3:  Leaf node вҶ’ scan for key 45 вҶ’ found вҶ’ follow pointer to data page
Step 4:  Read row from data page

Total I/O: 3-4 page reads (tree height + data page)
vs. Full table scan: potentially thousands of page reads
Complexity
  • B+Tree lookup: O(log_B N) where B is the branching factor (~100-500), N is row count
  • A table with 100 million rows needs only 3-4 levels of B+Tree вҶ’ 3-4 disk reads
  • Full table scan: O(N) вҖ” reads every page

Clustered vs. Non-ClusteredвҖӢ

Clustered Index (InnoDB Primary Key):

в”Ӯ  Only ONE clustered index per table         в”Ӯ

Non-Clustered (Secondary) Index:
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ  B+Tree leaves store KEY + POINTER to row   в”Ӯ
в”Ӯ  Pointer = PK value (InnoDB) or RID (heap)  в”Ӯ
в”Ӯ  Multiple secondary indexes per table       в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
Important: Secondary Index Lookup in InnoDB

In InnoDB, a secondary index lookup is a two-step process:

  1. Search the secondary index B+Tree вҶ’ get the primary key value
  2. Search the primary key (clustered) B+Tree вҶ’ get the actual row

This is called a bookmark lookup or double lookup. This is why covering indexes (which skip step 2) can dramatically improve performance.

Composite Index and the Leftmost Prefix RuleвҖӢ

A composite index on (a, b, c) can satisfy queries on:

  • вң… WHERE a = ?
  • вң… WHERE a = ? AND b = ?
  • вң… WHERE a = ? AND b = ? AND c = ?
  • вң… WHERE a = ? ORDER BY b
  • вқҢ WHERE b = ? вҖ” skips the leftmost column
  • вқҢ WHERE b = ? AND c = ? вҖ” skips a
  • вҡ пёҸ WHERE a = ? AND c = ? вҖ” uses index for a, but c requires a scan within that subset
-- This composite index...
CREATE INDEX idx_name ON orders (customer_id, order_date, status);

-- ...is effectively THREE indexes in one:
-- (customer_id)
-- (customer_id, order_date)
-- (customer_id, order_date, status)

Covering IndexвҖӢ

When an index contains all columns referenced in a query, the database can answer the query entirely from the index without touching the table вҖ” an index-only scan.

-- Query
SELECT customer_id, order_date, total
FROM orders
WHERE customer_id = 42 AND order_date > '2024-01-01';

-- Covering index (includes the SELECT column 'total')
CREATE INDEX idx_covering ON orders (customer_id, order_date, total);
-- Now the query never touches the heap вҖ” much faster!

Partial Index (PostgreSQL)вҖӢ

Index only the rows you care about вҖ” smaller index, faster maintenance.

CREATE INDEX idx_active_users ON users (email)
WHERE active = true;
-- Only indexes active users вҖ” much smaller than a full index
-- Only useful for queries that include WHERE active = true

Why B+Tree Dominates

B+Tree is preferred because:

  1. Higher fanout in internal nodes (no data pointers) вҶ’ shallower tree вҶ’ fewer I/O
  2. Linked leaf nodes вҶ’ efficient range scans without backtracking
  3. Predictable performance вҶ’ every lookup traverses the same depth

рҹ“Ӣ When Indexes Help vs. HurtвҖӢ

вң… Indexes Help WhenвҖӢ

ScenarioWhy
WHERE clause filtering on indexed columnsAvoids full table scan
JOIN conditionsSpeeds up nested loop joins
ORDER BY on indexed columnsAvoids filesort
GROUP BY on indexed columnsCan use index for grouping
SELECT DISTINCTIndex provides sorted unique values

Covering queriesIndex-only scan, no table access

вқҢ Indexes Hurt WhenвҖӢ

ScenarioWhy
Small tables (< few thousand rows)Full scan is faster than index traversal
High write workloadsEvery INSERT/UPDATE/DELETE must update all indexes
Low selectivity columnsBoolean or status columns with few distinct values
Frequent bulk loadsIndex maintenance slows down batch inserts
Over-indexingToo many indexes waste space and slow writes
Columns rarely queriedIndex maintenance cost with no read benefit
Rule of Thumb

An index is useful when it filters out most rows. If a query returns > 10-20% of the table, the optimizer may choose a full scan over an index scan because sequential I/O is faster than random I/O.

вҡЎ Query Optimization TechniquesвҖӢ

1. Use Covering IndexesвҖӢ

-- Before: Index Scan + heap lookup
CREATE INDEX idx_cust ON orders (customer_id);

-- After: Index Only Scan (no heap access)
CREATE INDEX idx_cust_covering ON orders (customer_id, order_date, total);

2. Avoid Functions on Indexed ColumnsвҖӢ

-- вқҢ Bad: function on column prevents index use (SARGability violation)
SELECT * FROM users WHERE YEAR(created_at) = 2024;

-- вң… Good: range predicate uses index
SELECT * FROM users WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';

3. Use EXISTS Instead of IN for Correlated SubqueriesвҖӢ

-- вқҢ Slower with large subquery results
SELECT * FROM orders WHERE customer_id IN (SELECT id FROM customers WHERE active = true);

-- вң… Faster: EXISTS stops at first match
SELECT * FROM orders o
WHERE EXISTS (SELECT 1 FROM customers c WHERE c.id = o.customer_id AND c.active = true);

4. Pagination with Keyset (Seek) Instead of OFFSETвҖӢ

-- вқҢ OFFSET scans and discards rows (O(offset + limit))
SELECT * FROM orders ORDER BY id LIMIT 20 OFFSET 100000;

-- вң… Keyset pagination (O(limit) вҖ” constant cost)
SELECT * FROM orders WHERE id > 100000 ORDER BY id LIMIT 20;

5. Batch Operations Instead of Row-by-RowвҖӢ

-- вқҢ N separate INSERT statements
INSERT INTO logs (msg) VALUES ('a');
INSERT INTO logs (msg) VALUES ('b');
INSERT INTO logs (msg) VALUES ('c');

-- вң… Single batch INSERT
INSERT INTO logs (msg) VALUES ('a'), ('b'), ('c');

рҹ“Ҳ Index Selectivity & CardinalityвҖӢ

Cardinality = number of distinct values in a column. Selectivity = cardinality / total rows (range 0 to 1).

ColumnCardinalitySelectivityGood Index?
user_id (PK)1,000,0001.0вң… Excellent
email999,900~1.0вң… Excellent
city5000.0005вҡ пёҸ Depends on query
status3 (active/inactive/banned)0.000003вқҢ Too low
is_deleted2 (true/false)0.000002вқҢ Use partial index
Composite Index Column Order

Put the most selective column first in composite indexes:

-- If customer_id is more selective than status:
CREATE INDEX idx_orders ON orders (customer_id, status);  -- вң… Good
CREATE INDEX idx_orders ON orders (status, customer_id);  -- вқҢ Less effective

Exception: if you always filter by status first in your queries, adapt to your query patterns.

рҹҡ« Common Anti-PatternsвҖӢ

1. SELECT *вҖӢ

-- вқҢ Fetches all columns, can't use covering indexes, wastes bandwidth
SELECT * FROM orders WHERE customer_id = 42;

-- вң… Select only needed columns
SELECT order_id, order_date, total FROM orders WHERE customer_id = 42;

2. N+1 Query ProblemвҖӢ

-- вқҢ 1 query to get orders, then N queries to get each customer
SELECT * FROM orders;                           -- 1 query
-- For each order:
SELECT * FROM customers WHERE id = ?;          -- N queries

-- вң… Single JOIN
SELECT o.*, c.name FROM orders o
JOIN customers c ON o.customer_id = c.id;       -- 1 query

3. Implicit Type ConversionвҖӢ

-- вқҢ phone_number is VARCHAR but compared to INT вҶ’ index not used
SELECT * FROM users WHERE phone_number = 1234567890;

-- вң… Match the type
SELECT * FROM users WHERE phone_number = '1234567890';

4. OR Conditions on Different ColumnsвҖӢ

-- вқҢ Cannot use a single index efficiently
SELECT * FROM users WHERE email = 'a@b.com' OR phone = '555-1234';

-- вң… Use UNION (each branch uses its own index)
SELECT * FROM users WHERE email = 'a@b.com'
UNION
SELECT * FROM users WHERE phone = '555-1234';

5. LIKE with Leading WildcardвҖӢ

-- вқҢ Leading wildcard prevents index use
SELECT * FROM products WHERE name LIKE '%widget%';

-- вң… Use full-text search for this pattern
SELECT * FROM products WHERE to_tsvector('english', name) @@ to_tsquery('widget');

рҹҺҜ Interview Questions on IndexingвҖӢ

Q1: Your query is slow. Walk me through how you'd diagnose and fix it.вҖӢ

Answer framework:

  1. Run EXPLAIN ANALYZE вҖ” identify scan type, row estimates, actual times
  2. Check for full table scans вҖ” is there a missing index?
  3. Check for SARGability вҖ” are functions applied to indexed columns?
  4. Check selectivity вҖ” is the index selective enough to be useful?
  5. Check for N+1 вҖ” is the application issuing too many queries?
  6. Consider covering index вҖ” can we avoid the heap lookup?
  7. Check statistics вҖ” are table statistics stale? Run ANALYZE.
  8. Check for lock contention вҖ” is the query waiting on locks?

Q2: What is the difference between a clustered and non-clustered index?вҖӢ

A clustered index determines the physical storage order of rows. There is only one per table (the primary key in InnoDB). A non-clustered (secondary) index is a separate structure that stores index keys + pointers (primary key values in InnoDB) to the actual rows. Secondary lookups require a second traversal of the clustered index.

Q3: When would you NOT create an index?вҖӢ

  • Column has very low cardinality (boolean, status with 2-3 values)
  • Table is small (< few thousand rows)
  • Table is write-heavy and rarely queried by that column
  • Column is frequently updated (index maintenance overhead)
  • You already have too many indexes on the table

Q4: What is a covering index and why does it matter?вҖӢ

A covering index includes all columns referenced in a query (SELECT, WHERE, JOIN, ORDER BY). The database answers the query entirely from the index without accessing the table heap вҖ” an Index Only Scan. This eliminates random I/O to the heap, which can be 2-10x faster.
